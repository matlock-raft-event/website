# Matlock Raft Event — Website

The marketing site and CMS for [The Matlock Raft Event](https://www.matlockraftevent.co.uk), a charity raft race held every Boxing Day on the River Derwent in Matlock, Derbyshire, raising funds for the RNLI.

- **Live site:** https://www.matlockraftevent.co.uk
- **Studio (CMS):** https://www.matlockraftevent.co.uk/studio
- **Sanity dashboard:** https://www.sanity.io/manage/project/6m6e8mul

## Stack

| Layer | Tech |
|---|---|
| Site framework | [Astro 6](https://astro.build) — static site generation |
| UI | [React 19](https://react.dev) (client islands), [Tailwind CSS 4](https://tailwindcss.com), [shadcn/ui](https://ui.shadcn.com) on [Base UI](https://base-ui.com) + [Vaul](https://vaul.emilkowal.ski) |
| Language | TypeScript 5.6 |
| CMS | [Sanity v5](https://www.sanity.io) — schemas + Studio in this repo, content served from the Sanity Content Lake CDN |
| Rich text | [@portabletext/react](https://github.com/portabletext/react-portabletext) |
| Icons | [Phosphor Icons](https://phosphoricons.com) + [Iconify](https://iconify.design) |
| Package manager | [pnpm 9](https://pnpm.io) |

## How the site is built

Astro pages run at **build time** in the repo root (`src/pages/*.astro`). Each page fetches the CMS data it needs from Sanity in its frontmatter, then passes typed props into a React island:

```ts
// src/pages/about.astro
const about = await sanityClient.fetch<AboutQueryResult>(aboutQuery);
```

```astro
<AboutPage client:load about={about} />
```

The React tree under `page-content/` and `sections/` is purely presentational. **No client-side Sanity requests in production** — the HTML ships with content baked in, then React hydrates for interactivity (drawer, accordion, etc.).

GROQ queries live in [`src/lib/queries.ts`](src/lib/queries.ts) wrapped in `defineQuery`. Schema-aware types are generated from them via `pnpm sanity:typegen` and committed at [`src/lib/sanity.types.ts`](src/lib/sanity.types.ts).

## Local development

Prerequisites: Node 22+, pnpm 9.

```bash
pnpm install
pnpm dev          # http://localhost:4321
```

The Studio is available at http://localhost:4321/studio. You'll need a Sanity account invited to the [project](https://www.sanity.io/manage/project/6m6e8mul) to log in.

### Common commands

| Command | What it does |
|---|---|
| `pnpm dev` | Astro dev server with HMR |
| `pnpm build` | Production build to `dist/` (43+ static pages) |
| `pnpm preview` | Serve the built `dist/` locally |
| `pnpm typecheck` | `astro check` — TypeScript + Astro diagnostics |
| `pnpm sanity:schema` | Extract Sanity schema → `schema.json` |
| `pnpm sanity:typegen` | Extract schema **and** regenerate `src/lib/sanity.types.ts` from GROQ queries |

After any schema change in `studio/schemas/`, run `pnpm sanity:typegen` and commit the diff.

## CMS

Content is edited by visiting `/studio` and signing in with a Sanity account. The Studio is the same React app that's used standalone — it's just embedded as a client-only Astro route, so visitor pages never load any of it.

### Singletons

The following documents are configured as singletons in the structure builder — only one of each can exist, and the editor UI hides "New" / "Duplicate" actions for them:

- `summary` (homepage stats)
- `about` (about page bios + RNLI/DASAC blurbs)
- `contactInstructions` (contact page sections)
- `cookiesInfo` (cookies page content)

### Adding new editors

Invite a Sanity user via the [project members page](https://www.sanity.io/manage/project/6m6e8mul/members). Permissions inherit from the project's role config; the standard role for content editors is "Editor".

### CORS

The Studio authenticates against the Sanity API directly from the browser. Every origin that serves `/studio` must be whitelisted at [Sanity → API → CORS Origins](https://www.sanity.io/manage/project/6m6e8mul/api):

- `https://www.matlockraftevent.co.uk` (production)
- `http://localhost:4321` (dev)

If a new domain (preview deploy, staging) hosts the Studio, it must be added here or login will fail with a generic auth error.

## Deployment

Production builds run `pnpm build`, output to `dist/`, and ship as static HTML + assets + a code-split client-only `/studio` route. No SSR adapter is needed.

The hosting platform's build configuration should be:
- **Build command:** `pnpm build`
- **Publish directory:** `dist`
- **Node version:** 22+
- **Install command:** `pnpm install --frozen-lockfile`
